HTML

  • HTML (Hypertext Markup Language) is used to create the structure of the webpage
    • It usually includes text elements, image elements, header elements, and list elements
    • HTML elements work together and provide the main framework for the end user to navigate a webpage
    • HTML elements have opening and closing tags using angle brackets (< >)
  • The head element contains information about the webpage
    • Metadata is data about our webpage that the browser uses to effectively serve our webpage to users
  • The body element represents the visible content shown to the user
  • The head and meta tags are invvisible to the end user
    • The one excpetion to this is the browser's title bar or a page's tab
  • HTML attriubtes alwyays defined in beginning element, never the closing element
    • The space in between the HTML opening and closing tab is what defines the entire structure of a website
    • For more in depth list of tags, Go to Tag Lists
  • You can easily create a skeleton HTML file that has some basic written code in VSS by:
    • Creating a new index.html file
    • Go to the search bar at the top
    • Type in a single "!" and hit enter
    • The context menu should automatically show the code in the browser before you click on it anyways

Cascading Style Sheets (CSS)

  • CSS is a programming language used to define how a webpage content is presented to the user
    • CSS defines colors, aesthetics, visual layout, and organization
    • We use CSS to define style rules for specific elements of an HTML page and format them using different style options
    • Three major ways to style a webpage
      • Inline css-section
        • When a developer wants to style an element directly in the HTML file, it is called inline css-section
        • Inline styles apply to a specific HTML tag using style attribute with a CSS rule
        • Useful for quick and permanent changes but less flexible than external or internal style sheets
        • Actually possible to embed an entire stylesheet in the HTML file
          • Internal style sheets contains CSS rules for the webpage in the head section of the HTML file
            • Rules will only apply to that page and are able to configure CSS classes and IDs
            • A single change to the CSS rule will apply to all tagged page elements
      • External style sheets
        • An external CSS file is linked to the HTML file using the <link> tag in the head section
        • Allows for centralized styling across multiple HTML files
        • Keeps all CSS rules in a seperate file, making it easier to change designs
        • Modifications in the external CSS file will affect all linked HTML files
        • Enables easy maintenance and separation of concerns between HTML and CSS
      • Imported style sheets
        • Similar to external style sheets, imported style sheets are linked to the HTML file but using the @import rule in a style block
        • Imported CSS files are loaded after the main CSS file and can override styles
        • Provides flexibility to include different CSS files based on specific conditions or requirements

GIT + GITHUB

  • Git (Global Information Tracker) is a version control system that keeps track of all versions of your project
    • Git is great for tracking local changes on your PC
    • It is installed locally on your computer and lets you open a terminal (CLI) called bash/git bash
    • Can be used to clone, create new branches, switch to different branches (called "checkout"), etc.
    • Refer to the Git Command Cheat Sheet for some common commands to use
      • git init: Initializes a new Git repository in the current directory.
      • git clone <repository>: Copies a repository from a remote source to your local machine.
      • git add <file>: Adds a file to the staging area to be included in the next commit.
      • git commit -m "message": Creates a new commit with the changes in the staging area and adds a commit message.
      • git push: Uploads local commits to a remote repository.
      • git pull: Fetches changes from a remote repository and merges them into the current branch.
      • git branch: Lists all branches in the repository.
      • git checkout <branch>: Switches to a different branch.
      • git merge <branch>: Merges changes from a specified branch into the current branch.
    • Can be connected to a Github repo for version control and can push, pull, merge, pull and more to the chosen Github repository
      • Must first establish SSH keys from your local machine and connect them to your Github account online
      • Can be difficult to establish so please refer to the Bootcamp's instructions as well as on Github's official documents
      • Also, it's important to make sure that you are always connected to your Github account using:
        • git status: show modified filed in your working directory, staged for next commit
        • git config --list: Shows current settings from the config file
        • git config user.name: displays current registered username
        • git config user.email: shows current registered email
    • A Git local environment refers to your own computer or development machine where you have a local copy of a Git repo.
      • Essentially the environment where you do Git operations and make changes to your codebase
      • Open your Git repo through the CLI and then launch the repo with the command "code ." to open it in Visual Studio Code
      • It's a safe space to test code without touching the online Github repo. Once your code is tested, it can then be pushed to the main Github repo (preferably as a separate branch AND NOT the main branch)
  • Github is a cloud-based hosting service that lets us keep track of repositories (or "repo's")
    • Similar to Git but via the cloud
    • Allows collaboration from multiple individuals for version control and is paramount towards the Agile method in a corporate setting
      • Agile principles lead to flexibility, customer satisfaction, and efficient software development.
      • Some of the key principles:
        • Iterative and incremental development in small cycles (iterations/sprints)
        • Continuous collaboration between teams, stakeholders, and end-users
        • Customer-centric approach, prioritizing their needs and incorporating feedback
        • Embracing change and adapting plans and requirements as necessary
        • Self-organizing cross-functional teams taking collective responsibility
        • Focus on continuous improvement through retrospectives
        • Popular frameworks: Scrum, Kanban, Extreme Programming (XP)
      • Gives you the ability to create an online repo which can be private or public and lets you continue your work without having to be tied down to the same PC workstation.
        • I do believe creating new SSH keys is needed on each PC workstation so that you can be connected to your Github account from the git bash terminal
      • README files are used to pass along useful information to anyone who visits the repo on github
      • Licenses are also needed for each repo, especially if they're public because it lets others know what can be done with the repo legally.
      • Github offers a task management tool called Github issues
    • Git and Github branching
      • Default branch is usually called main or master
      • NEVER DO CODING IN THE MAIN BRANCH!!!
      • Instead, create a separate branch off of the main branch for code testing
        • Feature branches is what they are usually called but can be called other names at user discretion
        • Do this via the Git terminal. Terminal command for it is in the cheat sheet and on the bootcamp module with further instructions
      • In the feature branch, we make commits of all the changes we make to the code while working. Here is a general rundown of what to do in the Gitbash terminal
        1. Create a feature branch:
          1. git branch feature-branch
          2. git checkout feature-branch
        2. Then make changes in your local environment (work on your code, modify files, etc.)
        3. Stage the changes
          1. git add file1.js file2.js (this is just an example and not actual file names)
          2. To stage an entire directory, do 'git add .'
        4. Commit the changes
          1. Git commit -m "Implement feature XYZ"
        5. Repeat steps iii-iv while continuing to make changes, staging them, and creating commits as needed
        6. Push the changes
          1. When you're ready to share with others, use the following command below to push your commits
          2. Git push origin feature-branch
          3. The push will be sent to the github repo where you will create what is called a "pull request". Other members in the team will review your pull request. Once the branch is reviewed and accepted, it will merge into the main branch.
      • You might be wondering what are commits and pushing because I for sure was confused
        • Commits capture a specific state of the codebase at a given point in time
          • Each commit is unique with its own identifier, commit messages, any references to parent commit(s).
        • Commits show a linear or branching history, showing the progress of the codebase over time
      • How often do you do commits?
        • Varies from person to person and from project to project
        • Common practices:
          • Granularity
            • Commits should represent logical units of work. It's generally recommended to make commits that encapsulate a cohesive set of changes related to a specific task or feature. This ensures that each commit is focused and makes it easier to understand the history of changes.
          • Regularly
            • It's good practice to make commits regularly rather than accumulating a large number of changes before committing. Regular commits help in maintaining a clear history, facilitate collaboration, and allow for easier rollback if necessary.
          • Milestones or Completed Features
            • Sometimes made when huge changes such as implementing new features. A sort of checkpoint status as you will.
          • Atomic changes
            • A commit represents a single conceptual change. If a change spans multiple files or modifications, it's recommended to split them into separate commits for clarity and easier tracking.
          • Continuous Integration/Deployment
            • Commits trigger automated processes like builds, tests, and deployments. In such cases, making commits more frequently, ideally after each completed task or code change, can help in ensuring smooth integration and continuous delivery.
      • Well, what about Staging? What is that?
        • Staging is necessary and must be done before making a commit.
        • Well not necessary, but it ensures that the properly selected files are committed instead of the entire repo
        • Staging allows you to selectively choose which changes you want to include in your next commit
          • By changes, I mean manually selecting which files to be staged for commits
        • If staging isn't done, if you ran the "git commit" command, Git would commit all of the modified files in your local repo by default.
        • Can be VERY problematic if you accidentally committed other files that were not ready to be committed yet
        • Staging is the safeguard against accidentally committing unintended changes and gives developers the ability to consciously curate the contents of each commit.

JAVASCRIPT

  • Javascript is a high-level dynamic programming language that allows users to interact with websites.
  • Javascript is the tool that takes a static HTML/CSS webpage and makes it interactive.
  • Some JavaScript use cases:
    • Update displayed data behind the scenes.
    • Display or use a timer.
    • Validate user input and show error messages + attach error styles to elements.
    • Display and manage complex overlays.
    • Re-order, hide/show, or remove elements on the screen.
    • Keep (parts of) websites working, even if the internet connection is lost.
  • Core Syntax:
    • (Syntax is just the "grammar" of a programming language, or the rules of it.)
    • 1.)Variables and Values.
    • 2.)Arrays and Objects.
    • 3.)Functions.
    • 4.)Working with the "DOM".
    • 5.)Control Structures.
  • Values:
    • Text values in JavaScript are called "String" values.
    • Number values are also important.
  • Variables:
    • Variables are "labeled data containers" called "var" in syntax.
    • JavaScript code can use either single or double quotes. Be consistent and use one or the other (i.e., "" or '').
    • Each line of JavaScript code ends with a semicolon (;).
  • Control flow is the order in which a computer executes code in a script:
    • JavaScript code runs from the first statement at the top of the page down to the last statement at the bottom of the page.
    • This means you must be intentional about where you write code in the JavaScript file!
    • Conditional statements and loops control the order in which code is executed or if it is even executed at all.
    • Using conditional statements allows us to interrupt the control flow, thus bypassing this limitation.
  • Conditional statements:
    • Such as "if-else" or "switch," enable you to execute different blocks of code depending on whether a certain condition is true or false.
    • An "if" statement is a way of introducing decision-making into our code.
    • In JavaScript, the coding language will check to see if a condition we provide is truthy.
    • If it is, then it will do whatever action we asked it to do between the curly brackets.
    • If the condition is not truthy, then JavaScript will ignore what's written in the curly brackets.
    • Falsy statements are defined by the values: 0, -0, On, " ", null, undefined, NaN, or a Boolean data type with the value of false.
    • We can add more conditions to if statements by using "else if".
  • Loops:
    • Enable you to repeat a set of instructions multiple times.
    • There are different types of loops, such as "for," "while," and "do-while".
  • Arrays:
    • Stores a group of related data in a variable and uses a "for" loop to perform a task on each piece of data.
    • An array is a single variable that is used to hold a group of data.
    • A single array can store different types of data, such as strings, numbers, and so on.
    • To work with individual data items inside an array, we need to know the location of the data in the array. The location is identified with a unique number called an index.
  • "For" loops:
    • A for loop uses the predictable pattern of indices to perform a task on all the items in an array by allowing a single code block to be executed over and over.
    • It has three main statements:
      • The first statement determines the starting point for our loop.
      • The second statement is the condition.
      • The final statement is what allows the array to iterate over each item.
  • Functions:
    • A set of instructions that tells the computer how to perform a certain task.
    • Functions are different from for loops and conditional statements because functions do not automatically execute when the JavaScript file is run.
    • You must call a function to execute it.
